# Kolektif — Distributed Volunteer Compute Network for ML Researchers

> *"What if every idle GPU in every lab could be part of something bigger?"*

---

## Motivation

Machine learning research has a compute problem — and it disproportionately affects people who can least afford to solve it.

Large institutions and well-funded labs take GPU access for granted. But for independent researchers, students, and academics at underfunded universities, the reality is different: cloud compute is expensive, university HPC clusters are bureaucratic nightmares with long queue times, and personal hardware is simply not enough for serious ML workloads.

This project was built by someone who felt that pain firsthand. The question that started it was simple: *if a lab has ten RTX 5070s sitting idle overnight, why can't a researcher on the other side of the university — or the other side of the world — use them?*

Kolektif is the answer to that question. It is a volunteer-based, distributed compute network that allows ML researchers to share their idle hardware with others who need it, with zero manual configuration, full task isolation, and a blockchain audit trail for every computation.

---

## What Is Kolektif?

Kolektif is a pull-based distributed task execution system designed specifically for ML workloads. It connects **consumers** (researchers who need compute) with **providers** (anyone willing to contribute their idle GPU or CPU) through a lightweight orchestration layer.

A researcher uploads their project, specifies an entrypoint, and submits a task. Somewhere on the network, a provider's machine picks it up, runs it inside an isolated Docker container with pre-installed ML libraries, and uploads the results to cloud storage. The researcher downloads their results. The entire lifecycle is logged to a blockchain ledger.

No manual SSH. No shared credentials. No trust required between parties.

## Goals & Objectives

### Primary Goal
Democratize access to ML compute by creating a frictionless, volunteer-driven alternative to expensive cloud providers and inaccessible HPC clusters.

### Specific Objectives

**For Consumers (Researchers who need compute):**
- Submit ML tasks with a single file upload and an entrypoint specification
- Receive results automatically upon completion
- Track task status in real time through a web interface
- Pay nothing — the network is volunteer-powered

**For Providers (Researchers who contribute compute):**
- Contribute idle hardware with minimal setup (Docker Desktop required)
- Maintain full control — the provider agent runs tasks in complete isolation
- No risk of malicious code affecting the host system
- Hardware is automatically detected and registered

**For the Research Community:**
- Create a foundation for a decentralized, trust-minimized compute commons
- Demonstrate that volunteer compute can be practical and secure for ML workloads
- Provide verifiable, tamper-proof records of all computations via blockchain

## Architecture

Kolektif is composed of four main layers:

### 1. Orchestrator
The central coordination server. It maintains the task queue, manages provider registration, handles authentication, and serves the REST API. Built with Flask and Gunicorn, backed by a PostgreSQL database (NeonDB).

The orchestrator operates on a **pull-based model**: providers poll for tasks rather than having tasks pushed to them. This design choice means providers can join and leave the network freely without the orchestrator needing to track their availability in real time.

### 2. Provider Agent
A lightweight daemon that runs on any machine willing to contribute compute. It periodically polls the orchestrator for available tasks, pulls the task's Docker image, executes the workload in a fully isolated container, and uploads results directly to cloud storage via a pre-signed URL. The agent never receives direct credentials — all storage access is mediated through short-lived signed URLs generated by the orchestrator.

### 3. Web UI
A React-based frontend that allows researchers to upload projects, submit tasks, monitor status, and download results. Built with a focus on simplicity — the goal is to make distributed compute feel as easy as uploading a file.

### 4. Blockchain Ledger
Every task lifecycle event — submission, assignment, completion, failure — is recorded on-chain via a smart contract. This creates an immutable, verifiable audit trail of all computations on the network. The ledger layer is currently deployed on the home server where the 512MB memory constraint of the previous hosting environment no longer applies.

### Infrastructure
- **Orchestrator:** Self-hosted on a dedicated home server (HP EliteDesk 800 G4, i5-8500T, 8GB DDR4), exposed publicly via Cloudflare Tunnel
- **Storage:** Cloudflare R2 for task artifacts and results
- **Database:** NeonDB (PostgreSQL)
- **Public URL:** `https://api.kolektif.cloud`
- **Web UI:** `https://app.kolektif.cloud`

## How It Works — Task Lifecycle

```
Researcher uploads project (ZIP) + specifies entrypoint (e.g. main.py)
        │
        ▼
Orchestrator receives task → assigns task_id → status: QUEUED
        │
        ▼
Provider agent polls orchestrator → detects idle GPU → claims task
        │
        ▼
Orchestrator generates pre-signed R2 upload URL → status: RUNNING
        │
        ▼
Provider pulls Docker image → runs task in isolated container
        │
        ▼
Results uploaded directly to R2 via pre-signed URL
        │
        ▼
Provider reports completion → status: COMPLETED
        │
        ▼
Blockchain ledger records final status on-chain
        │
        ▼
Researcher downloads results via pre-signed download URL
```

## Security Design

Security was a first-class concern in the design of Kolektif, given that providers are running arbitrary code submitted by strangers.

- **Task Isolation:** Every task runs inside a Docker container. The container has no access to the host filesystem beyond what is explicitly mounted, and no network access beyond what the task requires.
- **API Authentication:** All provider and consumer API endpoints require authentication via API keys passed in request headers. Public endpoints are explicitly whitelisted.
- **Credential-Free Storage:** Providers never receive R2 credentials. They upload results via short-lived pre-signed URLs generated per-task. This means a compromised provider agent cannot access any other task's data.
- **Enrollment Tokens:** Providers join the network via short-lived enrollment tokens (15-minute expiry) rather than static credentials.
- **Blockchain Auditability:** The immutable on-chain record means task history cannot be retroactively altered by any party, including the orchestrator operator.

## Current Status

Kolektif is a working MVP. The following components are fully operational:

- ✅ Orchestrator with task queue, provider management, and full REST API
- ✅ Provider agent with GPU detection, Docker execution, and result upload
- ✅ Web UI with task submission, status tracking, and result download
- ✅ Cloudflare R2 integration for artifact storage
- ✅ Self-hosted infrastructure on dedicated hardware at `api.kolektif.cloud`
- ✅ Blockchain ledger layer (active — no longer limited by hosting constraints)

The following are planned for future development:

- 🔲 Hardware capability handshake — automatic detection and registration of provider GPU/RAM specs at enrollment
- 🔲 Task splitting — automatic decomposition of large workloads into parallelizable subtasks
- 🔲 Worker pool model — concurrent task dispatch to multiple providers simultaneously (currently FIFO queue)
- 🔲 Fault tolerance — automatic task reassignment if a provider drops mid-execution
- 🔲 Performance benchmarking — systematic latency and throughput evaluation across multi-node configurations

## Getting Started

### Running the Orchestrator

```bash
git clone https://github.com/ruasnv/matcha-distributed-hpc
cd matcha-distributed-hpc
cp .env.example .env
# Fill in your environment variables
docker compose -f docker-compose.prod.yml up --build -d
```

| Variable | Description |
|---|---|
| `DATABASE_URL` | PostgreSQL connection string |

| `LEDGER_PRIVATE_KEY` | Ledger private key |

## Limitations & Honest Reflections

Kolektif is not a replacement for a proper HPC cluster. It was not designed to be.

The current task queue is strictly FIFO with no parallelism across providers — if multiple providers are online simultaneously, they do not yet operate as a true worker pool. The system has no fault tolerance if a provider disconnects mid-task. And provider onboarding requires Docker Desktop, which creates friction for non-technical contributors.

These are known limitations and represent the primary directions for future work, not fundamental flaws in the architecture. The core design — pull-based orchestration, containerized execution, credential-free storage, blockchain provenance — is sound and scales.

---

## Why This Matters

The compute divide in ML research is real and growing. As foundation models become the baseline for serious research, the gap between well-resourced and under-resourced researchers widens. Kolektif is a small attempt to push against that.

The vision is a network where a researcher in Ankara can submit a training job and have it run on idle GPUs across university labs — without paying cloud prices, without waiting in HPC queues, and without trusting a centralized platform with their data or their code.

That vision is not fully realized yet. But the foundation is built, it is running, and it works.
